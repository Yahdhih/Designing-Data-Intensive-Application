### Partie 1 — Scan séquentiel vs index B-tree (15 min)

```sql
-- Créer une table avec 100 000 lignes
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    prix DECIMAL(10, 2),
    categorie VARCHAR(50),
    stock INTEGER
);

-- Insérer 100 000 produits avec des données réalistes
INSERT INTO produits (nom, prix, categorie, stock)
SELECT
    'Produit-' || generate_series,
    (random() * 999 + 1)::DECIMAL(10,2),
    CASE (random() * 4)::INTEGER
        WHEN 0 THEN 'Électronique'
        WHEN 1 THEN 'Vêtements'
        WHEN 2 THEN 'Alimentation'
        WHEN 3 THEN 'Sport'
        ELSE 'Autre'
    END,
    (random() * 1000)::INTEGER
FROM generate_series(1, 100000);

-- Forcer Postgres à mettre à jour les statistiques
ANALYZE produits;

-- ── Requête 1 : Sans index ──
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM produits WHERE id = 42000;
-- Observe : "Seq Scan" ou "Index Scan" ? Pourquoi ?

-- ── Requête 2 : Même requête, en forçant le seq scan ──
SET enable_indexscan = off;
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE id = 42000;
SET enable_indexscan = on;

-- ── Comparer les temps (buffers hit = pages depuis le cache) ──
```

### Partie 2 — Créer des index et observer l'impact (20 min)

```sql
-- ── Index B-tree sur prix ──
CREATE INDEX idx_prix_btree ON produits (prix);
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE prix = 150.00;

-- ── Range scan avec index B-tree ──
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE prix BETWEEN 100 AND 200
ORDER BY prix;
-- Observe : "Index Scan" + "Sort" ou "Index Only Scan" ?

-- ── Index hachage sur prix ──
CREATE INDEX idx_prix_hash ON produits USING HASH (prix);
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE prix = 150.00;
-- Index hash = O(1) pour l'égalité exacte

-- ── Range scan avec index hash — observation clé ──
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE prix BETWEEN 100 AND 200;
-- Observe : Postgres N'utilise PAS l'index hash pour le range scan !
-- Pourquoi ? L'index hash ne préserve pas l'ordre.

-- ── Index GIN pour la recherche dans les tableaux (démonstration) ──
ALTER TABLE produits ADD COLUMN tags TEXT[];
UPDATE produits SET tags = ARRAY['solde', 'nouveau']
WHERE id % 3 = 0;

CREATE INDEX idx_tags_gin ON produits USING GIN (tags);
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE tags @> ARRAY['solde'];

-- ── Index partiel ──
CREATE INDEX idx_stock_faible ON produits (stock)
WHERE stock < 10;
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE stock < 10 ORDER BY stock;
```

### Partie 3 — Lire et comprendre EXPLAIN (10 min)

**Décoder un plan EXPLAIN typique :**

```
Index Scan using idx_prix_btree on produits  (cost=0.42..8.44 rows=1 width=120)
  (actual time=0.089..0.091 rows=1 loops=1)
  Index Cond: (prix = 150.00)
  Buffers: shared hit=3
```

- `cost=0.42..8.44` : coût estimé (démarrage..total) en unités arbitraires
- `rows=1` : nb de lignes estimées (compare avec `actual ... rows=1`)
- `actual time=0.089..0.091 ms` : vrai temps mesuré
- `Buffers: shared hit=3` : 3 pages lues depuis le cache (pas depuis le disque)

**Teste les limites :**
```sql
-- Quand Postgres préfère un Seq Scan même avec un index ?
-- (si la requête retourne >5-10% des lignes, le Seq Scan est souvent plus rapide)
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM produits WHERE prix < 900;  -- 90% des lignes → Seq Scan probable
```

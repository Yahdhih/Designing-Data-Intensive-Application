## 🧪 Labo — 45 min

### Architecture du labo

```
[Postgres Primary :5432]  ──WAL streaming──→  [Postgres Replica :5433]
        ↑                                              ↑
  ecritures (app)                              lectures (read replica)
```

### Étape 1 — Lancer le compose avec replica Postgres (10 min)

Crée `labs/compose.replica.yml` (fichier dédié à ce labo) :

```yaml
version: "3.8"

services:
  pg-primary:
    image: postgres:16.3
    container_name: ddia-pg-primary
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: ddia
    command: >
      postgres
      -c wal_level=replica
      -c max_wal_senders=3
      -c wal_keep_size=64
      -c synchronous_commit=on
    ports:
      - "5432:5432"
    volumes:
      - pg_primary_data:/var/lib/postgresql/data
    networks:
      - ddia-net

  pg-replica:
    image: postgres:16.3
    container_name: ddia-pg-replica
    environment:
      PGPASSWORD: secret
    command: >
      bash -c "
        until pg_basebackup -h pg-primary -U postgres -D /var/lib/postgresql/data -P -R --wal-method=stream; do
          echo 'Attente du primary...'; sleep 2;
        done
        postgres
      "
    ports:
      - "5433:5432"
    depends_on:
      - pg-primary
    volumes:
      - pg_replica_data:/var/lib/postgresql/data
    networks:
      - ddia-net

volumes:
  pg_primary_data:
  pg_replica_data:

networks:
  ddia-net:
    driver: bridge
```

```bash
# Démarrer primary + replica
docker compose -f labs/compose.replica.yml up -d

# Attendre ~15s que pg_basebackup termine
docker logs ddia-pg-replica -f
# → Attendre "database system is ready to accept read only connections"
```

### Étape 2 — Observer le lag en temps réel (10 min)

```bash
# Sur le PRIMARY : voir les connexions de réplication
docker exec ddia-pg-primary psql -U postgres -c "
  SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn)::text AS lag_bytes
  FROM pg_stat_replication;
"
```

**Crée des données sur le primary :**

```bash
docker exec ddia-pg-primary psql -U postgres -d ddia -c "
  CREATE TABLE IF NOT EXISTS messages (
    id SERIAL PRIMARY KEY,
    content TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
  );
  INSERT INTO messages (content)
  SELECT 'message ' || generate_series(1, 10000);
"
```

**Vérifie que le replica est à jour :**

```bash
docker exec ddia-pg-replica psql -U postgres -d ddia -c "
  SELECT COUNT(*) FROM messages;
"
# → doit afficher 10000
```

### Étape 3 — Mesurer le lag de réplication (10 min)

```bash
# Méthode 1 : via pg_stat_replication (sur le primary)
docker exec ddia-pg-primary psql -U postgres -c "
  SELECT
    application_name,
    state,
    pg_current_wal_lsn() AS primary_lsn,
    replay_lsn,
    (pg_current_wal_lsn() - replay_lsn) AS lag_bytes,
    now() - pg_last_xact_replay_timestamp() AS lag_time
  FROM pg_stat_replication;
"

# Méthode 2 : sur le replica directement
docker exec ddia-pg-replica psql -U postgres -c "
  SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag,
    pg_is_in_recovery() AS is_replica;
"
```

### Étape 4 — Simuler une surcharge et observer le lag (10 min)

```bash
# Écriture intensive sur le primary (2000 INSERT/s pendant 5s)
docker exec ddia-pg-primary psql -U postgres -d ddia -c "
  INSERT INTO messages (content)
  SELECT 'burst-' || generate_series(1, 100000);
"

# Pendant l'insertion — surveille le lag dans un second terminal
docker exec ddia-pg-primary psql -U postgres -c "
  SELECT
    (pg_current_wal_lsn() - replay_lsn) / 1024 AS lag_kb
  FROM pg_stat_replication;
" 
# → répète 3 fois toutes les 2 secondes pour observer l'évolution
```

### Étape 5 — Simuler une panne du replica (5 min)

```bash
# Pauser le replica
docker pause ddia-pg-replica

# Écrire sur le primary (lag augmente)
docker exec ddia-pg-primary psql -U postgres -d ddia -c "
  INSERT INTO messages (content) VALUES ('pendant-panne');
"

# Relancer le replica
docker unpause ddia-pg-replica

# Observer le catch-up
docker exec ddia-pg-replica psql -U postgres -c "
  SELECT now() - pg_last_xact_replay_timestamp() AS lag_après_reprise;
"
```

**Note :** En mode `wal_level=replica` avec réplication asynchrone, le primary n'attend pas le replica → si le replica est lent, le lag peut accumuler.

### Nettoyage

```bash
docker compose -f labs/compose.replica.yml down -v
```

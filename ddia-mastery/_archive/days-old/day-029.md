### Thèmes à lire (dans cet ordre) :

**1. Motivation et configuration (5 min)**
- Pourquoi répliquer ? (Durabilité · Disponibilité · Latence)
- La configuration mono-leader : 1 leader + N followers

**2. Synchrone vs asynchrone (10 min)**
- Réplication **synchrone** : le leader attend la confirmation du follower → durabilité totale, mais que se passe-t-il si le follower est lent ?
- Réplication **asynchrone** : le leader confirme immédiatement → latence d'écriture faible, mais durabilité réduite
- **Semi-synchrone** (1 follower synchrone, les autres asynchrones) — compromis

**3. Comment les followers s'initialisent (5 min)**
- Snapshot + replay du log de réplication
- Lien avec le WAL de Ch.3 : c'est le *même* WAL !

**4. Panne d'un follower : catch-up recovery (5 min)**
- Le follower connaît son dernier offset → replay depuis cet offset
- Lien avec les offsets Kafka (principe identique)

**5. Failover du leader (10 min)**
- Détection de panne (timeout heartbeat)
- Élection d'un nouveau leader (qui ? → le follower le plus à jour)
- Problèmes du failover automatique :
  - Données perdues si réplication asynchrone (le nouveau leader ne les a pas)
  - Split-brain (2 nœuds se croient leader simultanément)
  - Timeout trop court → failover intempestif sous charge

**6. Mécanismes de logs de réplication (5 min)**
- **WAL shipping** : envoi du WAL brut → couplage avec la version du moteur de stockage
- **Logical log** : changements au niveau ligne (plus portable)
- **Trigger-based** : pour la réplication filtrée

## 💡 Connexion avec tes expériences

*Kafka Producer `acks=all` vs `acks=1` — maintenant tu sais pourquoi ces options existent :*
- `acks=1` = réplication asynchrone (leader confirme sans attendre les followers)
- `acks=all` = réplication synchrone (tous les in-sync replicas doivent confirmer)
- `acks=0` = fire-and-forget (le producteur n'attend aucune confirmation)

*PostgreSQL streaming replication = WAL shipping asynchrone par défaut → `synchronous_standby_names` pour le mode synchrone.*

### Partie 1 — Postgres (15 min)

Modélise un profil utilisateur (nom, email, postes de travail, compétences).

```bash
docker compose -f labs/compose.light.yml exec postgres psql -U ddia
```

```sql
-- Créer le schéma
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(200) UNIQUE NOT NULL,
    summary TEXT
);

CREATE TABLE positions (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    title VARCHAR(200),
    company VARCHAR(200),
    start_year INTEGER,
    end_year INTEGER
);

CREATE TABLE skills (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    skill_name VARCHAR(100)
);

-- Insérer des données
INSERT INTO users (name, email, summary)
VALUES ('Alice Dupont', 'alice@example.com', 'Ingénieure data passionnée');

INSERT INTO positions (user_id, title, company, start_year, end_year)
VALUES (1, 'Data Engineer', 'Société ABC', 2020, 2023),
       (1, 'Senior Data Engineer', 'Startup XYZ', 2023, NULL);

INSERT INTO skills (user_id, skill_name)
VALUES (1, 'Spark'), (1, 'Kafka'), (1, 'Python');

-- Récupérer le profil complet
SELECT u.name, u.email, p.title, p.company, s.skill_name
FROM users u
LEFT JOIN positions p ON p.user_id = u.id
LEFT JOIN skills s ON s.user_id = u.id
WHERE u.id = 1;

-- Observer le plan d'exécution
EXPLAIN (ANALYZE, FORMAT TEXT)
SELECT u.name, p.title
FROM users u JOIN positions p ON p.user_id = u.id
WHERE u.id = 1;
```

**Observe :** combien de tables impliquées pour un profil simple ? Quel est le plan d'exécution sans index ? Avec index ?

```javascript
// Créer la base
use ddia

// Insérer le même profil en document imbriqué
db.users.insertOne({
  name: "Alice Dupont",
  email: "alice@example.com",
  summary: "Ingénieure data passionnée",
  positions: [
    { title: "Data Engineer", company: "Société ABC", start_year: 2020, end_year: 2023 },
    { title: "Senior Data Engineer", company: "Startup XYZ", start_year: 2023, end_year: null }
  ],
  skills: ["Spark", "Kafka", "Python"]
})

// Récupérer le profil complet (1 seule requête, pas de jointure)
db.users.findOne({ email: "alice@example.com" })

// Filtrer par compétence
db.users.find({ skills: "Spark" })

// Ajouter un champ sans ALTER TABLE — pas besoin
db.users.updateOne(
  { email: "alice@example.com" },
  { $set: { github_url: "https://github.com/alice" } }
)
```

### Partie 3 — Comparaison (15 min)

**Questions à documenter dans `reference/notes/jour-008.md` :**

1. **Nombre de requêtes pour récupérer le profil complet :**
   - Postgres : ___ requêtes (ou 1 avec JOIN) — combien de lignes retournées ?
   - MongoDB : ___ requête — format ?

2. **Ajouter un champ `github_url` :**
   - Postgres : `ALTER TABLE users ADD COLUMN github_url VARCHAR(200);` — que se passe-t-il pour les lignes existantes ?
   - MongoDB : `$set` direct — pas d'ALTER TABLE, mais qu'est-ce que ça implique pour la cohérence ?

3. **Requête many-to-many :** Trouve tous les utilisateurs qui ont travaillé dans la même entreprise qu'Alice.
   - Postgres : JOIN + GROUP BY — possible mais…
   - MongoDB : très difficile sans `$lookup` (et `$lookup` est lent sur de gros datasets)

4. **Schema-on-read vs schema-on-write en pratique :**
   - Insérer `db.users.insertOne({ typo_name: "Bob" })` → MongoDB accepte silencieusement
   - Postgres : `INSERT INTO users (typo_name) VALUES ('Bob')` → erreur immédiate
   - Lequel préfères-tu pour un système de paiement ? Pourquoi ?

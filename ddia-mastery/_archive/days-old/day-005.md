### Points clés à retenir

1. **Impedance mismatch :** les données applicatives sont en objets (langage OOP) ; le stockage est en tables. La traduction objet↔table est une friction permanente → d'où les ORMs, mais qui ont leurs propres problèmes.

2. **Modèle document — avantages :**
   - Localité : tout le document chargé en une lecture (un utilisateur = un document, pas 5 jointures)
   - Schéma flexible (*schema-on-read* vs *schema-on-write*)
   - Correspond naturellement à la structure objet
   - Bon pour les données *one-to-many* (un utilisateur → plusieurs postes)

3. **Modèle relationnel — avantages :**
   - Jointures puissantes pour les *many-to-many*
   - Schéma strict → moins de bugs silencieux
   - Transactions ACID
   - Meilleur pour les données liées entre entités différentes (ex. commandes ↔ produits ↔ clients)

4. **Schema-on-read vs Schema-on-write :** NoSQL = schéma implicite dans le code (lu au moment de la lecture) · SQL = schéma explicite dans la base (imposé à l'écriture). Chacun a ses risques.

5. **Tendance 2026 :** les frontières s'estompent — Postgres supporte JSON/JSONB · MongoDB supporte les transactions multi-documents. Le choix dépend du *workload dominant*.

### 📌 Contexte 2026 (si 1ʳᵉ éd.)
La 2ᵉ éd. souligne davantage la **convergence** des modèles (SQL with JSON, MongoDB with ACID transactions). La question n'est plus « SQL ou NoSQL ? » mais « quel workload dominant ? ». Avec ton expérience OpenSearch — note que les documents JSON y sont indexés de manière inversée, un autre compromis.

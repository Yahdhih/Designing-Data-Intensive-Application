### Points clés à retenir

1. **Problèmes de JSON/XML :**
   - Aucune distinction entre entiers et flottants → `{"age": 30}` ou `{"age": 30.0}` ?
   - Pas de types binaires natifs → doit encoder en base64
   - Verbeux : les noms de champs répétés à chaque objet
   - Évolution de schéma non vérifiée → champ manquant = bug silencieux

2. **Protobuf / Thrift :**
   - Chaque champ a un **numéro de tag** (entier) au lieu d'un nom
   - Le décodeur identifie les champs par leur numéro, ignore ceux qu'il ne connaît pas
   - **Backward compat** : ajouter un nouveau champ → les anciens décodeurs l'ignorent (forward compat)
   - **Danger** : changer le numéro d'un champ = incompatibilité silencieuse

3. **Avro :**
   - Pas de numéros de tags → le schéma est séparé (writer's schema ≠ reader's schema)
   - Résolution de schéma à la lecture : mappe les champs par nom
   - Le schéma doit être accessible lors de la désérialisation → Schema Registry (Confluent)
   - Idéal pour Kafka (schéma centralisé, évolution contrôlée)

4. **Comparaison de taille (exemple `{user_id: 42, name: "Alice"}`) :**
   - JSON : ~30 bytes (les noms de champs répétés)
   - Protobuf : ~10 bytes (field tag + type + valeur)
   - Avro : ~8 bytes (pas de tags, schéma séparé)

5. **Lien Kafka :** Confluent Schema Registry stocke les schémas Avro/Protobuf → permet la résolution writer/reader → c'est pourquoi Kafka + Avro est une combinaison standard

### 📌 Contexte 2026 (si 1ʳᵉ éd.)
La 2ᵉ éd. couvre également **JSON Schema** comme moyen de valider JSON (schema-on-write partiel), et **Apache Arrow** (format colonne en mémoire) qui gagne en importance pour le traitement in-memory (DuckDB, Polars). Le principe de base reste le même.

## 📖 Synthèse Ch.6 — 35 min

### Partie 1 — Matrice de décision (15 min, de mémoire d'abord)

Remplis d'abord **sans regarder tes notes**, puis compare et corrige :

**Table 1 : Stratégie de partitionnement**

| Critère | Range | Hash | Consistent Hash + vnodes |
|---------|-------|------|--------------------------|
| Distribution des données | Variable (hot spots temporels) | Uniforme | Uniforme (avec vnodes) |
| Range scan | ✓ Efficace | ✗ Scatter | ✗ Scatter |
| Rééquilibrage (ajout nœud) | Manuel (split) | Catastrophique (`%N`) | Minimal (O(K/N)) |
| Hot spot possible | ✓ (plage récente) | ✓ (clé populaire) | ✓ (clé populaire, salting) |
| Exemples | HBase, Postgres RANGE, Bigtable | Cassandra (sans vnodes) | Cassandra (avec vnodes), Dynamo |

**Table 2 : Index secondaires**

| Critère | Local (scatter/gather) | Global (term-partitioned) |
|---------|------------------------|--------------------------|
| Écriture | Simple (1 partition) | Complexe (N partitions) |
| Lecture | Scatter (latence max) | Directe (1 partition) |
| Cohérence | Forte | Éventuelle (souvent async) |
| Exemples | MongoDB, Cassandra, Postgres local | DynamoDB GSI, Elasticsearch global |

**Table 3 : Routage des requêtes**

| Approche | Connaissance routing | Avantage | Inconvénient |
|----------|---------------------|----------|--------------|
| Client-side | Client (Redis Cluster) | Pas de proxy | Config complexe côté client |
| Routing tier | Proxy (HAProxy, CockroachDB) | Client simple | Proxy = SPOF, latence +1 hop |
| Any-node | Nœud redirige (Cassandra) | Transparent pour client | Latence +1 hop si mauvais nœud |

### Partie 2 — Arbre de décision Ch.6 (10 min)

Dessine dans `reference/notes/jour-048-decision-tree.md` :

```
Ai-je besoin de range scans sur la clé principale ?
├── Oui → Range partitioning (HBase, Postgres RANGE)
│   └── Attention : hot spot si clé = timestamp → préfixer avec hash
└── Non → Hash partitioning
    ├── Cluster fixe → hash % N (Kafka, Elasticsearch)
    └── Cluster dynamique → Consistent Hash + vnodes (Cassandra, Dynamo)

Ai-je des requêtes sur des attributs secondaires ?
├── Écriture prioritaire → Index local (scatter/gather à la lecture)
└── Lecture prioritaire → Index global (term-partitioned, async)

Exemple : Cassandra
  → Clé de partition = hash(partition_key)  [consistent hash]
  → Clé de clustering = range scan LOCAL (dans la partition)
  → Requête sans partition_key → scatter sur tous les nœuds
  → Solution : modéliser les données pour éviter les scans cross-partition
```

### Partie 3 — Connexion Ch.6 → Ch.7 (10 min)

Le partitionnement crée un nouveau problème : les **transactions distribuées**.

Si une transaction modifie 2 partitions (ex : transfert bancaire entre comptes sur 2 shards différents) :
- Partition A débite 100€ de Alice
- Partition B crédite 100€ de Bob
- Que se passe-t-il si la Partition B tombe en panne après que A a débité ?

→ C'est le problème des **transactions cross-partition** que Ch.7 résoudra.

| Problème | Solution Ch.5 | Solution Ch.6 | Combiné |
|----------|--------------|--------------|---------|
| 1 nœud pas assez disponible | Réplication | — | — |
| 1 nœud pas assez rapide (lecture) | Followers | — | Partitions × Replicas |
| 1 nœud pas assez grand (stockage) | — | Partitionnement | Partitions × Replicas |
| 1 nœud pas assez rapide (écriture) | — | Partitionnement | — |

**Formule à retenir :**
```
Infrastructure d'un système distribué =
  Partitions (Ch.6) × Replicas (Ch.5)
         ↓
  Chaque partition a RF copies
  RF copies = résistance aux pannes
  N partitions = scalabilité horizontale
```

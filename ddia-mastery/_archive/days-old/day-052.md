**1. Snapshot isolation (10 min)**

Idée centrale : chaque transaction voit un snapshot de la base au moment où elle a commencé. Les modifications committées par d'autres transactions après le début de la transaction sont *invisibles*.

```
T1 (début à t=10) : BEGIN
T2 (début à t=11) : BEGIN
T2 : UPDATE accounts SET balance = 50 WHERE id = 'Alice'
T2 : COMMIT  (t=12)
T1 : SELECT balance FROM accounts WHERE id = 'Alice'
     → voit toujours 100€ (snapshot de t=10, avant T2)
T1 : COMMIT  (t=13)
```

Avantage : les lectures *longues* (backup, analytics) ne bloquent pas les écritures.

Nommage : appelé « Repeatable Read » dans MySQL et Oracle, « Snapshot Isolation » dans Postgres et SQL Server. Les noms SQL sont malheureusement incohérents entre les BDD.

**2. MVCC — Multi-Version Concurrency Control (10 min)**

Implémentation standard de snapshot isolation.

Chaque ligne a **plusieurs versions** (une par transaction qui l'a modifiée) :
```
Row 'Alice.balance' :
  version tx_id=5  : 100€  (créée par la transaction 5)
  version tx_id=12 : 50€   (créée par la transaction 12)
```

Chaque transaction voit uniquement les versions dont `tx_id ≤ snapshot_ts`.

Règles de visibilité MVCC :
- T (tx courante) voit la version V si : `V.tx_id < T.start_tx_id` ET `V` est committée avant T.
- T ne voit pas les versions créées après T.start_tx_id.

Avantage : **readers don't block writers, writers don't block readers** — contrairement à 2PL.

**3. Write skew (15 min)**

Le problème le plus subtil de Ch.7. **Snapshot isolation ne l'empêche pas.**

Exemple des médecins de garde (DDIA classique) :
```
Invariant : au moins 1 médecin doit être disponible à tout moment.
État initial : Alice = disponible, Bob = disponible.

T1 (Alice) : SELECT COUNT(*) FROM doctors WHERE on_call = true  → 2
             (assez, je peux me libérer)
T2 (Bob)   : SELECT COUNT(*) FROM doctors WHERE on_call = true  → 2
             (assez, je peux me libérer)
T1 : UPDATE doctors SET on_call = false WHERE name = 'Alice'
T1 : COMMIT
T2 : UPDATE doctors SET on_call = false WHERE name = 'Bob'
T2 : COMMIT

Résultat : 0 médecin disponible. Invariant violé.
```

Pourquoi snapshot isolation ne suffit pas :
- T1 et T2 lisent un snapshot cohérent (2 médecins)
- T1 écrit sur une ligne que T2 ne lit pas (Alice ≠ Bob) → pas de conflict write-write
- T2 écrit sur une ligne que T1 ne lit pas
- Mais les 2 écritures, prises ensemble, violent l'invariant global

Ce pattern est : **lire l'ensemble → décider → écrire une valeur dépendante de la lecture**.

Autres exemples de write skew :
- Réservation d'une salle (2 clients réservent la même salle pour le même créneau)
- Allocation d'username (2 utilisateurs prennent le même pseudo)
- Système de rendez-vous médicaux

Solutions :
- `SELECT ... FOR UPDATE` sur toutes les lignes lues → verrouille l'ensemble
- Materialized conflict : créer une ligne de « verrou » explicite
- Serializable isolation (J.53)

*Ferme le livre — reformule write skew en 3 phrases + 1 exemple.*

**4. Phantom reads (5 min)**

Variante du write skew où la lecture porte sur une **plage** (range) plutôt qu'un ensemble fixe de lignes.

```
T1 : SELECT * FROM bookings WHERE room = 123 AND start < '2024-12-25' AND end > '2024-12-20'
     → 0 résultats (créneau libre)
T2 : INSERT INTO bookings (room, start, end) VALUES (123, '2024-12-20', '2024-12-25')
T2 : COMMIT
T1 : INSERT INTO bookings (room, start, end) VALUES (123, '2024-12-20', '2024-12-25')
T1 : COMMIT
→ 2 réservations pour la même chambre au même moment !
```

Le phantom = T2 insère une ligne que T1 n'a pas vue (mais qui existait logiquement dans sa plage de lecture).

Solution : **predicate locks** (verrouiller la plage de la requête) → seule vraie solution est Serializable.

### 2 cartes Anki (5 min)

```tsv
Q: Qu'est-ce que le write skew ? Pourquoi snapshot isolation ne l'empêche pas ?
R: Write skew : 2 txns lisent un ensemble → chacune modifie un sous-ensemble différent
   → la combinaison viole un invariant global.
   Snapshot isolation ne l'empêche pas car les écritures portent sur des lignes différentes
   (pas de conflict write-write détecté) mais la lecture partagée est stale pour l'une des txns.
Tags: ch07::transactions::write-skew

Q: Comment MVCC implémente-t-il snapshot isolation ? (mécanisme clé)
R: Chaque modification crée une nouvelle VERSION de la ligne (avec un tx_id).
   Chaque transaction lit uniquement les versions dont tx_id ≤ snapshot_ts (start de la txn).
   Résultat : readers don't block writers, writers don't block readers.
   Coût : stockage de plusieurs versions → vacuum/garbage collection nécessaire (Postgres).
Tags: ch07::transactions::mvcc
```

## 📖 Lecture — 40 min

📌 **Localise la section :** `reference/toc.md` → Ch.7 "Serializability"

**1. La définition de sérialisabilité (5 min)**

Une exécution de N transactions concurrentes est *sérialisable* si son résultat est *identique* à une exécution *séquentielle* des mêmes transactions (dans un ordre quelconque).

C'est la garantie la plus forte possible. Elle interdit *toutes* les anomalies : dirty reads, lost updates, write skew, phantom reads.

Mais elle a un coût — deux implémentations, deux compromis.

*Reformule en 2 phrases.*

**2. Exécution sérielle réelle (5 min)**

L'approche la plus simple : 1 seule transaction à la fois.
- 1 thread d'exécution, 1 CPU → pas de concurrence → pas d'anomalie possible
- Conditions requises : les données tiennent en RAM, les transactions sont courtes (< 1ms), pas de latence I/O pendant la transaction (stored procedures)
- Utilisé par : VoltDB, Redis (mono-thread), H-Store

Limitation : pas scalable (1 CPU).

*Reformule en 2 phrases.*

**3. Two-Phase Locking (2PL) (15 min)**

L'approche classique (utilisée depuis les années 1970).

Principe : verrous **partagés** (shared) et **exclusifs** (exclusive) sur chaque ligne :
- Un lecteur acquiert un verrou *partagé* → plusieurs lecteurs peuvent coexister
- Un écrivain acquiert un verrou *exclusif* → bloque tous les autres accès

Les 2 phases :
1. **Phase d'acquisition** : acquérir les verrous au fur et à mesure des lectures/écritures
2. **Phase de libération** : libérer tous les verrous au COMMIT / ROLLBACK

→ Pas de verrou relâché pendant l'exécution de la transaction.

**Deadlock :**
```
T1 : verrouille ligne A → veut ligne B (attend T2)
T2 : verrouille ligne B → veut ligne A (attend T1)
→ Deadlock ! La BDD détecte et rollback une des deux transactions.
```

**Predicate locks :** Pour les phantom reads, il faut verrouiller non pas des lignes, mais une *condition* (`WHERE room = 123 AND date = '...'`). Coûteux → la plupart des BDD utilisent des **index-range locks** (approximation).

Performance : 2PL avec beaucoup de concurrence → contention de verrous → latences élevées. C'est pourquoi peu de systèmes utilisent Serializable 2PL par défaut.

*Reformule en 3 phrases.*

**4. Serializable Snapshot Isolation (SSI) (15 min)**

Inventé en 2008 (Cahill et al.). Utilisé par Postgres depuis 9.1 (`ISOLATION LEVEL SERIALIZABLE`).

Idée : **optimiste**. Les transactions s'exécutent comme en snapshot isolation (sans verrous) et on *détecte* les conflits au commit.

Comment détecter le write skew ?

SSI surveille les **dépendances read-write** entre transactions :
- T1 a lu une valeur → T2 l'a modifiée (après la lecture de T1) → *T2 a rendu stale la lecture de T1*
- Si cette dépendance forme un cycle → write skew potentiel → T1 ou T2 est abortée

Ce que SSI surveille :
- Quand T1 lit une version d'une ligne, SSI note « T1 a lu cette ligne »
- Quand T2 modifie cette ligne, SSI note « T2 a invalidé la lecture de T1 »
- Si T2 commite avant T1, SSI vérifie si cela crée un conflit → abort T1 si nécessaire

Avantages vs 2PL :
- Pas de verrous → readers ne bloquent pas writers
- Moins de deadlocks
- Meilleure performance sous faible contention

Inconvénient : aborts plus fréquents sous haute contention (l'application doit retry).

*Reformule en 3 phrases.*

## 🧠 Consolidation — 20 min

### Comparaison 2PL vs SSI (10 min)

Dans `reference/tradeoffs.md` — complète la section Ch.7 Serializable :

| Dimension | 2PL | SSI |
|-----------|-----|-----|
| Philosophie | Pessimiste (verrouille préventivement) | Optimiste (détecte après coup) |
| Readers bloquent writers | Oui (verrou partagé) | Non |
| Writers bloquent readers | Oui (verrou exclusif) | Non |
| Deadlock possible | Oui | Non (aborts à la place) |
| Performance basse contention | Bonne | Bonne |
| Performance haute contention | Mauvaise (contention) | Bonne (mais aborts) |
| Implémentation | Postgres (sans SSI), MySQL | Postgres ≥ 9.1 (BEGIN LEVEL SERIALIZABLE) |

### 2 cartes Anki (5 min)

```tsv
Q: Quelle est la différence fondamentale entre 2PL et SSI pour implémenter la sérialisabilité ?
R: 2PL = pessimiste : verrouille les données lues/écrites avant de les utiliser → bloque les autres.
   SSI = optimiste : s'exécute sans verrous, détecte les conflits au commit → abort si conflit.
   2PL : deadlocks possibles, mauvaise perf sous haute contention.
   SSI : aborts possibles, bonne perf sous faible contention (Postgres ≥ 9.1).
Tags: ch07::transactions::serializable

Q: Comment SSI détecte-t-il le write skew du problème des médecins de garde ?
R: SSI surveille les dépendances read-write entre transactions :
   T1 lit on_call_count → T2 modifie on_call_count → SSI note que T2 a invalidé la lecture de T1.
   Si T2 commite, SSI détecte un cycle de dépendances → abort T1 ou T2.
   L'application doit retry la transaction abortée.
Tags: ch07::transactions::serializable::ssi
```

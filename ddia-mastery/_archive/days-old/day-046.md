## 📖 Lecture — 35 min

📌 **Localise la section :** `reference/toc.md` → Ch.6 "Partitionnement et index secondaires"

### Thèmes à lire

**1. Le problème des index secondaires avec le partitionnement (5 min)**

Supposons que tu partitionnes les utilisateurs par `hash(user_id)`.  
Requête simple : `SELECT * WHERE user_id = 42` → hash(42) → partition 3 → 1 nœud.  
Requête secondaire : `SELECT * WHERE country = 'France'` → ?  
→ Le pays n'est pas la clé de partition → impossible de savoir dans quelle partition les utilisateurs français se trouvent → il faut demander à *toutes* les partitions.

Deux approches pour les index secondaires :

**2. Index secondaire document-partitioned (local) — scatter/gather (15 min)**

Chaque partition maintient son propre index secondaire (index *local*).  
- Écriture de `user_id=42, country=France` → va en partition 3 → l'index local de P3 enregistre `France → [42]`
- Lecture `WHERE country = France` → envoyer la requête à *toutes* les partitions → récupérer les résultats → agréger (**scatter/gather**)

Avantage : écriture simple (1 seule partition mise à jour).  
Inconvénient : lecture lente (scatter/gather sur N partitions → latence = max(latences) → sensible aux nœuds lents).

Utilisé par : MongoDB, Cassandra, Riak, Elasticsearch (indices locaux par shard).

*Ferme le livre — reformule en 3 phrases.*

**3. Index secondaire term-partitioned (global) (10 min)**

L'index global est lui-même partitionné, mais par **terme** (valeur indexée), pas par document.  
- Partition A de l'index → termes commençant par a-m (ex : `Australia`, `France`)
- Partition B de l'index → termes n-z

Écriture de `user_id=42, country=France` → met à jour P3 (données) + P_index(France) (index).  
Lecture `WHERE country = France` → 1 seule partition d'index → liste des user_ids → éventuellement scatter pour récupérer les documents.

Avantage : lecture plus efficace (pas de scatter sur toutes les partitions).  
Inconvénient : écriture plus coûteuse (met à jour plusieurs partitions) + souvent asynchrone → index peut être périmé quelques secondes.

Utilisé par : DynamoDB Global Secondary Index, Elasticsearch (global replica), Cassandra (Materialized Views).

*Ferme le livre — reformule en 3 phrases.*

**4. Routage des requêtes (5 min)**

Comment un client sait-il vers quel nœud envoyer sa requête ?

Trois approches :
- **Client-side routing** : le client connaît le mapping partitions → nœuds (ex : Redis Cluster)
- **Routing tier** : un proxy intermédiaire connaît le mapping et route (ex : CockroachDB)
- **Any-node routing** : le client envoie vers n'importe quel nœud, qui redirige si nécessaire (Cassandra, Riak)

Qui maintient le mapping partition → nœud ?  
- **ZooKeeper** ou **etcd** : source de vérité pour le cluster state (Kafka l'utilisait jusqu'à KRaft, Elasticsearch l'utilise)
- **Gossip protocol** : les nœuds se parlent entre eux (Cassandra)

### Tableau comparatif des approches d'index (10 min)

| Aspect | Local (scatter/gather) | Global (term-partitioned) |

| Écriture | 1 partition | Multiple (données + index) |

### Questions de consolidation (10 min)

1. **Scatter/gather et tail latency :** Si chaque partition répond en p99=50ms, mais tu as 20 partitions, quelle est la p99 de la requête scatter/gather ? Pourquoi est-ce problématique pour les latences de queue ?

2. **Index global et cohérence :** Tu écris `user_id=42, country=France`. L'index global est mis à jour *asynchroniquement*. Si tu fais immédiatement `WHERE country=France`, verras-tu user_id=42 ? Quel concept Ch.5 décrit cette situation ?

3. **Routage et ZooKeeper :** Dans Kafka (que tu connais), qui maintient le mapping `topic:partition → leader` ? Que se passe-t-il quand un leader change (failover) ? Comment les producers Kafka connaissent-ils le nouveau leader ?

Q: Qu'est-ce que le scatter/gather dans un index secondaire document-partitioned ?
R: Chaque partition a son propre index local.
   Lecture WHERE country=France → envoi de la requête à TOUTES les partitions
   → chacune renvoie ses résultats locaux → agrégation côté coordinateur.
   Coût : latence = max(latences) → sensible aux nœuds lents (tail latency amplification).
Tags: ch06::partitionnement::index-secondaire

Q: Avantage principal de l'index term-partitioned (global) vs local (scatter/gather) ?
R: L'index global est partitionné par terme (valeur indexée).
   Lecture WHERE country=France → 1 seule partition d'index (contient France) → liste des IDs.
   Avantage : évite le scatter/gather → lecture plus rapide.
   Inconvénient : écriture met à jour plusieurs partitions → souvent asynchrone → index périmé.

### Points clés à retenir

1. **Modèle graphe — quand ?** : Quand les données ont de nombreuses relations many-to-many et qu'on veut traverser des liens (« amis d'amis », détection de fraude, systèmes de recommandation, routage). Les jointures SQL explosent en profondeur de graphe.

2. **Property graph** (Neo4j) : nœuds + arêtes, chacun avec des propriétés libres. **Triple store / RDF** (Jena, AllegroGraph) : sujet + prédicat + objet — plus proche du web sémantique.

3. **Déclaratif** (SQL, Cypher, Datalog) :
   - Tu décris *quoi* obtenir
   - Le moteur décide *comment* → peut optimiser, paralléliser, changer l'ordre sans changer le résultat
   - Exemple : l'optimizer SQL peut choisir entre 5 plans d'exécution différents

4. **Impératif** (Java, Python, MapReduce) :
   - Tu décris *comment* obtenir le résultat, étape par étape
   - Pas d'optimisation automatique possible
   - Plus flexible pour la logique métier complexe

5. **MapReduce** : hybride étrange — le *framework* est déclaratif (tu définis map/reduce) mais le *code* des fonctions est impératif. Moins d'optimisations possibles que SQL pur.

### 📌 Contexte 2026 (si 1ʳᵉ éd.)
La 2ᵉ éd. couvre davantage **GraphQL** comme langage de requête orienté API (à ne pas confondre avec les bases graphes). Et les **LLMs** changent la donne : des requêtes en langage naturel → SQL/Cypher générés automatiquement. Mais les *modèles* sous-jacents restent les mêmes.

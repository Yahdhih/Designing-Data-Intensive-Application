## 🛠️ Exercice « Conçois X » — 45 min

### Contexte du problème

**Système de réservation de billets d'avion**

- **Contrainte fondamentale :** Un siège ne peut être réservé que par *1 passager* (no double-booking)
- **Charge :** 10 000 vols actifs, chaque vol = 180 sièges, pics de réservation à 5 000 req/s
- **Opérations :**
  - `reserve_seat(flight_id, seat_id, passenger_id)` → succès ou échec
  - `list_available(flight_id)` → liste des sièges disponibles
  - `cancel_reservation(reservation_id)` → libérer un siège
- **Contrainte supplémentaire :** La réservation doit tenir < 10s de paiement (timeout)

---

### Partie 1 — Schéma SQL (8 min)

Dans `reference/notes/jour-060-design.md`, propose un schéma minimal :

```sql
-- Table principale des sièges
CREATE TABLE seats (
    flight_id    INT,
    seat_id      TEXT,       -- ex: "12A"
    status       TEXT CHECK (status IN ('available', 'reserved')),
    reserved_by  INT,        -- passenger_id (NULL si available)
    reserved_at  TIMESTAMPTZ,
    PRIMARY KEY (flight_id, seat_id)
);

-- Table des réservations (pour l'historique + annulation)
CREATE TABLE reservations (
    reservation_id BIGSERIAL PRIMARY KEY,
    flight_id      INT,
    seat_id        TEXT,
    passenger_id   INT,
    created_at     TIMESTAMPTZ DEFAULT now(),
    expires_at     TIMESTAMPTZ,  -- pour les réservations temporaires (10s)
    status         TEXT CHECK (status IN ('pending', 'confirmed', 'cancelled'))
);
```

**Questions sur le schéma :**
1. Y a-t-il une contrainte UNIQUE manquante qui empêche le double-booking au niveau BDD ?
2. Ajoute la contrainte `UNIQUE (flight_id, seat_id) WHERE status = 'reserved'` — est-ce possible en Postgres ? (indice : index partiel)

### Partie 2 — Quelle anomalie le double-booking représente-t-il ? (7 min)

Classe le problème dans la taxonomie Ch.7 :

```
Scénario :
  T1 : SELECT * FROM seats WHERE flight_id=42 AND seat_id='12A' → status='available'
  T2 : SELECT * FROM seats WHERE flight_id=42 AND seat_id='12A' → status='available'
  T1 : UPDATE seats SET status='reserved', reserved_by=101 WHERE ...
  T1 : INSERT INTO reservations (flight_id, seat_id, passenger_id) VALUES (42, '12A', 101)
  T1 : COMMIT
  T2 : UPDATE seats SET status='reserved', reserved_by=102 WHERE ...
  T2 : INSERT INTO reservations ...
  T2 : COMMIT → double-booking !
```

Questions :
- S'agit-il d'un lost update, d'un write skew ou d'un phantom read ? Justifie.
- Quel niveau d'isolation suffit à éviter ça ? Pourquoi pas Repeatable Read seul ?

### Partie 3 — 3 approches de protection (15 min)

Propose, justifie, et compare les 3 approches suivantes :

**Approche A — Contrainte UNIQUE + INSERT**
```sql
-- Transformer la réservation en INSERT sur une table de réservations avec UNIQUE
-- Au lieu de UPDATE status='reserved', on INSERT et on laisse la contrainte rejeter les doublons
CREATE UNIQUE INDEX ON reservations(flight_id, seat_id) WHERE status != 'cancelled';

-- Transaction :
BEGIN;
INSERT INTO reservations (flight_id, seat_id, passenger_id, status)
VALUES (42, '12A', 102, 'confirmed');
-- → Si doublon : UNIQUE violation → ERROR → client retry avec autre siège
COMMIT;
```

**Approche B — SELECT FOR UPDATE**
```sql
BEGIN;
SELECT * FROM seats
WHERE flight_id = 42 AND seat_id = '12A'
FOR UPDATE;  -- verrou exclusif
-- Si status = 'reserved' → ROLLBACK
-- Si status = 'available' → UPDATE + INSERT + COMMIT
COMMIT;
```

**Approche C — BEGIN ISOLATION LEVEL SERIALIZABLE**
```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- Lire + écrire normalement
-- Postgres SSI détecte les conflits et abort si nécessaire
-- Client doit retry
COMMIT;
```

Pour chaque approche, note dans `reference/notes/jour-060-design.md` :

| Approche | Anomalie évitée | Blocage ? | Retry nécessaire ? | Perfo sous 5k req/s |
|----------|----------------|-----------|-------------------|---------------------|
| A : UNIQUE | | | | |
| B : FOR UPDATE | | | | |
| C : SERIALIZABLE (SSI) | | | | |

### Partie 4 — Gestion du timeout 10s (15 min)

Problème : si le passager réserve mais ne paie pas dans les 10s, le siège doit être libéré.

**Approche 1 — EXPIRES_AT + cron cleanup**
```sql
-- Réserver avec expiration
INSERT INTO reservations (flight_id, seat_id, passenger_id, status, expires_at)
VALUES (42, '12A', 102, 'pending', now() + interval '10 seconds');

-- Cron job toutes les 30s :
UPDATE seats SET status = 'available', reserved_by = NULL
WHERE (flight_id, seat_id) IN (
    SELECT flight_id, seat_id FROM reservations
    WHERE status = 'pending' AND expires_at < now()
);
UPDATE reservations SET status = 'cancelled'
WHERE status = 'pending' AND expires_at < now();
```

**Questions :**
1. Pendant les 10s, le siège est marqué 'reserved'. Un autre client peut-il quand même lire ce siège comme 'available' ? (quel niveau d'isolation ?)
2. Quel problème se pose si le cron ne tourne pas pendant 5 minutes (bug) ?
3. Alternative : utiliser une file de message Kafka avec un TTL → est-ce mieux ? Pourquoi ?

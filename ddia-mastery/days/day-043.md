### Thèmes à lire

**1. Pourquoi partitionner ? (5 min)**

La réplication donne N copies d'un même jeu de données → scalabilité lecture.  
Le partitionnement divise les données en shards → scalabilité stockage + écriture.

En pratique :
- Chaque partition est un sous-ensemble indépendant des données
- Chaque partition peut être répliquée (RF copies de chaque partition)
- Requête sur 1 partition = 1 nœud → parallélisme naturel

**2. Partitionnement par plage de clés (10 min)**

Exemple : clé = timestamp (logs), plage = [2024-01-01, 2024-06-30]

Avantages :
- Range scan efficace : `SELECT * WHERE ts BETWEEN t1 AND t2` → 1 partition
- Clés triées → scans séquentiels (lecture rapide sur disque)

Inconvénient — Hot spot :
- Si la clé est le timestamp → toutes les écritures vont dans la partition « aujourd'hui »
- Les autres partitions (anciennes dates) sont inactives
- Solution : préfixer la clé avec un hash → `hash(sensor_id) + timestamp`

*Ferme le livre — reformule en 3 phrases.*

**3. Partitionnement par hachage de clé (10 min)**

- `partition_id = hash(key) % N_partitions`
- Distribution uniforme des clés → plus de hot spot sur une plage
- Inconvénient : les range scans sont impossibles (`WHERE user_id BETWEEN 100 AND 200` → scatter sur toutes les partitions)

Exemples :
- Cassandra : `partition_key` haché → MurmurHash → partition
- MongoDB : champ de `shard key` haché
- Kafka : `hash(key) % n_partitions` (tu le connais)

Cas particulier — hot spot *même avec le hachage* :  
Si une clé unique est très populaire (célébrité avec 30M followers), toutes les écritures pour cette clé vont dans 1 partition.  
Solution : salting → ajouter un suffixe aléatoire (`key + random_suffix(0..N)`) → répartir sur N partitions → les lectures doivent lire les N partitions et agréger.

*Ferme le livre — reformule en 3 phrases.*

**4. Partitionnement skewed et hot spots (10 min)**

DDIA fournit plusieurs exemples concrets de hot spots :
- Réseau social : tweet d'une célébrité → 1 clé, des millions d'écritures/seconde
- E-commerce : lancement d'un produit → 1 clé article, des milliers de commandes/s
- Jeux vidéo : une zone du jeu très populaire

Stratégies :
- Détection de hot spots par le monitoring (p99 latence sur 1 partition)
- Salting applicatif pour les clés populaires connues
- Request splitting : le client envoie la requête à N partitions

*Ferme le livre — reformule en 3 phrases.*

**5. Rebalancing — introduction (5 min)**

Que se passe-t-il quand on ajoute un nœud ?
- `hash(key) % N` → si N change, presque toutes les clés bougent → catastrophe
- `hash(key) % (N+1)` ≠ `hash(key) % N` pour ~N/(N+1) des clés

Solution : *consistent hashing* (demain, J.44) ou *fixed partitioning* (Kafka).

### Expérience 1 — Lost update avec Read Committed (10 min)

Exécute les commandes **en alternance** dans les 2 sessions, dans l'ordre indiqué :

```
Session 1                              Session 2
─────────────────────────────────────  ─────────────────────────────────────
BEGIN;
SELECT balance FROM accounts
WHERE id = 1;   → 1000
                                       BEGIN;
                                       SELECT balance FROM accounts
                                       WHERE id = 1;   → 1000
UPDATE accounts SET balance = 1100
WHERE id = 1;
COMMIT;
                                       UPDATE accounts SET balance = 1050
                                       WHERE id = 1;
                                       COMMIT;
```

```sql
-- Après : quelle est la valeur finale ?
SELECT * FROM accounts;
-- Attendu : 1050 (le +100 de S1 est perdu !)
```

**Observation :** En Read Committed, le lost update est possible.

---

### Expérience 2 — Lost update résolu avec `FOR UPDATE` (10 min)

```
Session 1                              Session 2
─────────────────────────────────────  ─────────────────────────────────────
-- Reset
UPDATE accounts SET balance = 1000 WHERE id = 1;

BEGIN;
SELECT balance FROM accounts
WHERE id = 1 FOR UPDATE;   → 1000
                                       BEGIN;
                                       SELECT balance FROM accounts
                                       WHERE id = 1 FOR UPDATE;
                                       -- ↑ BLOQUÉ (attend que S1 commite)
UPDATE accounts SET balance = 1100
WHERE id = 1;
COMMIT;
                                       -- S2 se débloque, relit : 1100
                                       UPDATE accounts SET balance = 1200
                                       WHERE id = 1;
                                       COMMIT;
```

```sql
SELECT * FROM accounts;
-- Attendu : 1200 (1000 + 100 + 100 = les 2 incréments préservés ✓)
```

### Expérience 3 — Write skew avec Repeatable Read (10 min)

```
-- Reset
UPDATE doctors SET on_call = true;

Session 1 (Alice)                      Session 2 (Bob)
─────────────────────────────────────  ─────────────────────────────────────
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM doctors
WHERE on_call = true;   → 2
                                       BEGIN ISOLATION LEVEL REPEATABLE READ;
                                       SELECT COUNT(*) FROM doctors
                                       WHERE on_call = true;   → 2
UPDATE doctors
SET on_call = false
WHERE name = 'Alice';
COMMIT;
                                       UPDATE doctors
                                       SET on_call = false
                                       WHERE name = 'Bob';
                                       COMMIT;
```

```sql
SELECT * FROM doctors;
-- Attendu : les 2 on_call = false → 0 médecin ! Invariant violé.
```

**Observation :** Write skew possible même en Repeatable Read / Snapshot Isolation.

---

### Expérience 4 — Write skew résolu avec Serializable (SSI) (10 min)

```
-- Reset
UPDATE doctors SET on_call = true;

Session 1 (Alice)                      Session 2 (Bob)
─────────────────────────────────────  ─────────────────────────────────────
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT COUNT(*) FROM doctors
WHERE on_call = true;   → 2
                                       BEGIN ISOLATION LEVEL SERIALIZABLE;
                                       SELECT COUNT(*) FROM doctors
                                       WHERE on_call = true;   → 2
UPDATE doctors SET on_call = false
WHERE name = 'Alice';
COMMIT;
                                       UPDATE doctors SET on_call = false
                                       WHERE name = 'Bob';
                                       COMMIT;
                                       -- ↑ ERREUR attendue :
                                       -- ERROR: could not serialize access due to
                                       -- concurrent update
```

```sql
SELECT * FROM doctors;
-- Attendu : Alice=false, Bob=true → 1 médecin ✓ (SSI a aborted Bob)
```

**Observation :** SSI a détecté le cycle de dépendances et aborté la transaction de Bob. Bob doit retry.

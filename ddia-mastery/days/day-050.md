### Thèmes à lire

**1. Pourquoi les transactions ? (5 min)**

Sans transactions, un bug ou une panne à mi-chemin d'une opération peut laisser la base dans un état incohérent.

Exemple : transfert bancaire sans transaction :
```
1. SELECT balance FROM accounts WHERE id = 'Alice' → 100€
2. UPDATE accounts SET balance = 0 WHERE id = 'Alice'  ← panne ICI
3. UPDATE accounts SET balance = 200 WHERE id = 'Bob'   ← jamais exécuté
```
→ 100€ ont disparu.

La transaction enveloppe ces 3 opérations : si une échoue, tout est rollback.

**2. Atomicité (10 min)**

Définition DDIA : *toutes les écritures d'une transaction réussissent ou toutes sont annulées* — il n'y a pas d'état intermédiaire visible.

Mécanisme : le WAL (déjà vu en Ch.3 !) permet de savoir si une transaction était complète ou non au moment d'une panne. Si la panne survient avant le `COMMIT`, le WAL permet de rollback.

**Note importante :** Atomicité ≠ concurrence. Atomicité garantit que *ta* transaction est tout-ou-rien. Elle ne dit rien sur ce que *d'autres* transactions voient. C'est le rôle de l'Isolation.

*Reformule en 3 phrases.*

**3. Cohérence (5 min)**

Définition DDIA : *le « C » de ACID est la lettre la plus faible* — c'est une propriété **applicative**, pas une propriété de la base. La base ne peut pas vérifier que `solde_Alice + solde_Bob = constante`. C'est l'application qui doit écrire des transactions cohérentes.

DDIA critique le terme « C » comme un « tour de passe-passe » pour former l'acronyme « ACID ».

**4. Isolation (10 min)**

C'est la propriété la plus complexe et la plus importante de Ch.7.

Définition : *les transactions s'exécutent comme si elles étaient seules* — elles ne « voient » pas les transactions concurrentes en cours.

Mais « comme si elles étaient seules » a plusieurs niveaux de rigueur :
- **Serializable** : les transactions paraissent s'exécuter l'une après l'autre (le plus fort)
- **Snapshot isolation** : chaque transaction voit un snapshot cohérent
- **Read committed** : on ne lit que les données committées (le plus commun)
- **Read uncommitted** : on peut lire des données non committées (dangereux)

→ Chaque niveau laisse passer certaines anomalies. C'est ce que J.51-52 explorera en détail.

*Reformule en 3 phrases.*

**5. Durabilité (5 min)**

Définition : *une transaction committée survit aux pannes matérielles*.

Mécanismes :
- WAL : écrit sur disque *avant* de confirmer le commit
- Réplication : la transaction est committée sur au moins 1 replica
- En pratique : combinaison des deux

Nuance : la « durabilité parfaite » n'existe pas — un datacenter peut brûler. La durabilité est relative à la configuration.

**6. Transactions simple-objet vs multi-objets (5 min)**

- **Simple-objet :** mise à jour d'un seul document/ligne → atomicité triviale
- **Multi-objets :** mise à jour de plusieurs lignes ou tables → besoin de transactions explicites

Cas où les transactions multi-objets sont indispensables :
- Dénormalisation : mettre à jour un compteur + les entrées qui le composent
- Index secondaires : mettre à jour le document + l'index
- Contraintes de cohérence : `solde >= 0` avec débit sur 2 comptes

Q: Quelle est la différence entre atomicité et isolation dans ACID ?
R: Atomicité = ta propre transaction est tout-ou-rien (même sans concurrence).

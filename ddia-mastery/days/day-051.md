## 📖 Lecture — 35 min

📌 **Localise la section :** `reference/toc.md` → Ch.7 "Niveaux d'isolation faibles"

**1. Read uncommitted (5 min)**

Le niveau le plus faible. Autorise les **dirty reads** :
- T1 lit la valeur écrite par T2, mais T2 n'a pas encore commité
- Si T2 est rollback, T1 a lu une valeur qui n'a jamais existé

Exemple :
```
T1 : BEGIN
T2 : BEGIN
T2 : UPDATE accounts SET balance = 0 WHERE id = 'Alice'  ← pas encore commité
T1 : SELECT balance FROM accounts WHERE id = 'Alice'  → voit 0 (dirty read)
T2 : ROLLBACK  ← annulé, Alice n'a jamais eu 0€
```

→ Pratiquement inutilisable en production. Aucune BDD sérieuse ne l'utilise par défaut.

*Ferme le livre — reformule en 2 phrases.*

**2. Read committed (15 min)**

Le niveau le plus utilisé en production (défaut Oracle, SQL Server, Postgres avec certaines configurations non-SSI).

**Garanties :**
- **Pas de dirty reads** : on ne lit que les données committées
- **Pas de dirty writes** : on n'écrase pas des données non-committées

**Comment l'implémenter :**
- Pour les dirty writes : verrou au niveau ligne (row-level lock) — T2 attend que T1 commite avant d'écrire sur la même ligne
- Pour les dirty reads : maintenir l'ancienne valeur (committed) et la nouvelle (en cours) → retourner l'ancienne jusqu'au commit de T1

**Anomalie encore possible avec Read Committed — Non-repeatable read :**
```
T1 : BEGIN
T1 : SELECT balance FROM accounts WHERE id = 'Alice'  → 100€
T2 : BEGIN
T2 : UPDATE accounts SET balance = 50 WHERE id = 'Alice'
T2 : COMMIT
T1 : SELECT balance FROM accounts WHERE id = 'Alice'  → 50€ ← différent !
T1 : COMMIT
```
T1 voit 2 valeurs différentes pour la même ligne dans la même transaction → violation de cohérence interne à la transaction.

*Ferme le livre — reformule en 3 phrases.*

**3. Lost update — une anomalie subtile (10 min)**

Le **lost update** n'est pas un problème de dirty read — il se produit même avec Read Committed :

Pattern : read-modify-write concurrent
```
T1 : v = SELECT counter FROM stats  → v = 42
T2 : v = SELECT counter FROM stats  → v = 42
T1 : UPDATE stats SET counter = v + 1  → 43
T2 : UPDATE stats SET counter = v + 1  → 43 (écrase T1 !)
```
Résultat : le compteur vaut 43 au lieu de 44. L'incrément de T1 a été perdu.

Solutions :
- **Atomic write** : `UPDATE stats SET counter = counter + 1` → pas de read-modify-write → atomique
- **Explicit locking** : `SELECT ... FOR UPDATE` → T2 attend T1
- **Compare-and-set (CAS)** : `UPDATE stats SET counter = 43 WHERE counter = 42` → retour 0 rows si collision → retry

*Ferme le livre — reformule en 3 phrases.*

**4. Portée de Read Committed (5 min)**

Postgres défaut = **Read Committed** mais avec un twist : chaque `SELECT` voit son propre snapshot (au moment du SELECT). Ce n'est pas exactement « Snapshot Isolation » — c'est une version plus faible.

Le défaut de Postgres pour les transactions explicites (`BEGIN; ...; COMMIT`) est Read Committed. Pour Snapshot Isolation → `BEGIN ISOLATION LEVEL REPEATABLE READ`.

### 3 cartes Anki (5 min)

```tsv
Q: Qu'est-ce qu'un dirty read ? Quel niveau d'isolation l'interdit ?
R: Dirty read = lire une valeur écrite par une transaction qui n'a pas encore commité.
   Si cette transaction rollback, on a lu une valeur qui n'a jamais existé.
   Interdit dès Read Committed (le niveau par défaut dans la plupart des BDD).
Tags: ch07::transactions::isolation

Q: Qu'est-ce qu'un lost update ? Donne 3 solutions.
R: Lost update = 2 txns lisent la même valeur, l'incrémentent, l'écrivent → 1 seul incrément survit.
   Solution 1 : atomic write (UPDATE count = count + 1).
   Solution 2 : SELECT FOR UPDATE (verrou exclusif).
   Solution 3 : compare-and-set (WHERE count = valeur_lue → retry si 0 rows).
Tags: ch07::transactions::isolation::lost-update

Q: Le niveau Read Committed empêche-t-il les non-repeatable reads ? Pourquoi ?
R: Non. Read Committed garantit seulement qu'on ne lit que des données committées.
   Entre 2 SELECT dans la même transaction, une autre transaction peut commiter → valeur change.
   → Même requête retourne 2 résultats différents dans la même transaction.
   Solution : Repeatable Read / Snapshot Isolation.
Tags: ch07::transactions::isolation

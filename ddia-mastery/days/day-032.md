### Thèmes à lire

**1. Pourquoi multi-leader ? (5 min)**

Cas d'usage réels :
- **Multi-datacenter** : 1 leader par datacenter pour réduire la latence d'écriture (le client écrit dans son datacenter local)
- **Apps hors-ligne** (calendrier, notes) : le smartphone est son propre leader → synchro quand reconnecté
- **Édition collaborative** (Google Docs) : chaque curseur est un « leader »

Avantage vs inconvénient clé :
- Avantage : écriture locale dans chaque datacenter → latence d'écriture faible
- Inconvénient majeur : **conflits d'écriture** (la même donnée modifiée par 2 leaders simultanément)

*Ferme le livre — reformule en 3 phrases.*

**2. Conflits d'écriture (10 min)**

Exemple concret : Alice modifie le titre d'un document depuis Paris, Bob le modifie depuis Tokyo en même temps. Quand les 2 leaders se synchronisent, ils ont des valeurs différentes pour le même champ.

Comment les détecter :
- Mono-leader : pas de conflit possible (un seul nœud accepte les écritures)
- Multi-leader : le conflit est détecté à la réception de la réplication depuis l'autre leader

Stratégies de résolution (lis-les dans l'ordre, comprends les compromis) :
- **Last-Write-Wins (LWW)** : garder la valeur avec le timestamp le plus récent → perte de données silencieuse si les horloges sont désynchronisées
- **Fusion explicite** : les deux valeurs sont conservées, l'application résout (`v1: "Alice", v2: "Bob"` → l'app affiche les deux)
- **CRDTs** (Conflict-free Replicated Data Types) : structures de données conçues pour fusionner sans conflit (compteurs, ensembles, texte)
- **Résolution par l'application** (callback) : le développeur écrit la logique de résolution

*Ferme le livre — reformule en 3 phrases.*

**3. Topologies de réplication multi-leader (10 min)**

3 topologies :
```
Circulaire :  A → B → C → A  (anneau)
En étoile :   A → B ← C · A → D ...
Tout-à-tout : A ↔ B ↔ C ↔ A (chaque leader réplique vers tous les autres)
```

Problèmes :
- Circulaire/étoile : un seul nœud en panne peut couper la chaîne de réplication
- Tout-à-tout : les writes peuvent arriver dans le mauvais ordre (un write dépendant d'un autre peut arriver en avance) → problème de **causalité**

*Ferme le livre — reformule en 3 phrases.*

**4. Lien avec les vector clocks (5 min)**

La causalité dans les systèmes multi-leader nécessite de savoir *quel write précède quel autre*. C'est là qu'interviennent les **vector clocks** (Ch.8) — abordés en détail plus tard.

Pour l'instant : note que LWW résout le problème de manière simple mais *perd des données*. Les CRDTs le résolvent sans perte, mais pour des structures de données spécifiques seulement.

Q: Pourquoi Last-Write-Wins (LWW) peut-il causer des pertes de données silencieuses ?
R: LWW utilise le timestamp de l'horloge murale (wall clock) pour choisir le "gagnant".

Q: Qu'est-ce qu'un CRDT et pourquoi évite-t-il les conflits par construction ?
R: Conflict-free Replicated Data Type : structure de données dont les opérations
   commutent (ordre n'importe pas) et sont idempotentes.
   Ex : compteur G (increment-only) → fusionner = prendre le max de chaque nœud.
Tags: ch05::replication::multi-leader::crdt

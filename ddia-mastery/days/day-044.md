**1. Le problème du rééquilibrage avec `hash % N` (5 min)**

Si N passe de 10 à 11 :  
`hash(key) % 10` ≠ `hash(key) % 11` pour la quasi-totalité des clés.  
→ On déplace ~90% des données → tempête de réseau, cluster inutilisable pendant le rééquilibrage.

**2. Hachage cohérent — l'anneau (15 min)**

Idée : l'espace de hachage est un anneau [0, 2^32).  
- Chaque **nœud** est haché → position fixe sur l'anneau
- Chaque **clé** est hachée → position sur l'anneau → va vers le premier nœud dans le sens horaire

Dessine l'anneau avec 3 nœuds (A, B, C) et 6 clés :
```
        0°
        A
  330°    ╲  30°
        ↑   k1
   C  300°    60°
        │
   k6   │  k2
  270°──┼──90°
        │
   k5   │  k3
  240°    120°
        B
  210°    150°
        180°
        k4
```
→ k1, k2 → B (premier nœud dans le sens horaire depuis k1, k2)
→ k3, k4 → C
→ k5, k6 → A

**Ajout d'un nœud D entre A et B :**
- D prend uniquement les clés qui étaient assignées à B et qui sont entre A et D
- Toutes les autres clés restent où elles sont
- Movement minimum : O(K/N) clés bougent (K = total clés, N = nœuds)

**Suppression du nœud B :**
- Les clés de B vont au nœud suivant (C)
- Toutes les autres clés restent où elles sont

*Ferme le livre — redessine l'anneau de mémoire avec 3 nœuds et 1 ajout.*

**3. Virtual nodes (vnodes) (10 min)**

Problème avec l'anneau simple :
- La distribution peut être inégale (coup de chance dans le placement des nœuds)
- Un nœud physique fort et un nœud faible reçoivent la même charge

Solution : chaque nœud physique est representé par V positions sur l'anneau (vnodes).  
- `V = 150` (Cassandra par défaut)
- Nœud A → positions {hash(A-0), hash(A-1), ..., hash(A-149)}
- Distribution plus uniforme (loi des grands nombres)
- Nœud fort → plus de vnodes → plus de données

Avantage du rééquilibrage avec vnodes :  
Quand un nœud rejoint le cluster, il ne prend qu'une fraction de vnodes de *chaque* nœud existant → charge du rééquilibrage distribuée sur tout le cluster (pas 1 seul nœud qui reçoit tout).

*Ferme le livre — explique le rôle des vnodes en 3 phrases.*

**4. Stratégies de rééquilibrage (5 min)**

- **Fixed partitioning** (Kafka, Elasticsearch) : N partitions fixes au départ. Quand un nœud rejoint, il reçoit quelques partitions entières. Simple, mais il faut bien choisir N dès le départ.
- **Dynamic partitioning** (HBase, Cassandra) : les partitions se divisent automatiquement si elles dépassent un seuil. Flexible mais plus complexe.
- **Manual** : administrateur décide quelles partitions migrent. Plus de contrôle, moins d'automatisation.

### 3 cartes Anki (10 min)

```tsv
Q: Comment le hachage cohérent réduit-il le nombre de clés à déplacer lors d'un rééquilibrage ?
R: Les nœuds et les clés sont placés sur un anneau [0, 2^32).
   Quand un nœud est ajouté, il ne prend que les clés situées entre lui et son prédécesseur.
   → Seulement O(K/N) clés bougent (vs presque toutes avec hash % N).
Tags: ch06::partitionnement::consistent-hash

Q: Quel problème les virtual nodes (vnodes) résolvent-ils dans l'anneau de hachage cohérent ?
R: Sans vnodes, la distribution peut être inégale (placement aléatoire → variance).
   Avec V vnodes par nœud physique, la distribution tend vers l'uniformité (grands nombres).
   Avantage extra : le rééquilibrage est distribué sur tous les nœuds (pas un seul qui reçoit tout).
Tags: ch06::partitionnement::consistent-hash::vnodes

Q: Quelle est la différence entre fixed partitioning (Kafka) et dynamic partitioning (Cassandra) ?
R: Fixed : N partitions définies à la création. Un nœud reçoit des partitions entières.
   Simple, il faut bien choisir N (trop petit = bottleneck, trop grand = overhead).
   Dynamic : les partitions se divisent quand elles dépassent un seuil.
   Flexible, adapté aux données dont la taille varie dans le temps.
Tags: ch06::partitionnement::rebalancing
```

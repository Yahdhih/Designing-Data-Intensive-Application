### Points clés à retenir

1. **Log append-only + hash index (Bitcask) :**
   - Écriture : toujours append → toujours séquentielle → très rapide
   - Lecture : index hachage en RAM {clé → byte_offset} → O(1)
   - Contrainte fatale : toutes les clés doivent tenir en RAM
   - Compaction : fusionner les segments, supprimer les doublons

2. **SSTables :**
   - Clés triées dans chaque segment → merge efficace (comme tri-fusion)
   - Index creux : 1 entrée d'index pour K entrées → beaucoup moins de RAM
   - Range scan possible (« toutes les clés entre A et B »)
   - Segments immuables → compaction sans verrous

3. **Le filtre de Bloom :**
   - Structure probabiliste : tester si une clé *pourrait* être dans un SSTable
   - Faux positifs possibles (on lit pour rien), faux négatifs impossibles
   - Permet de skip des SSTables sans les lire → économise des I/O

4. **LSM-tree complet :**
   - memtable (en RAM, trié) → flush vers SSTable L0 → compaction vers L1, L2…
   - Écriture : toujours dans la memtable (rapide) + WAL (crash recovery)
   - Lecture : cherche dans la memtable, puis L0, L1… → potentiellement lente

### 📌 Contexte 2026 (si 1ʳᵉ éd.)
Les LSM-trees sont devenus la structure dominante pour le stockage objet cloud-natif (S3, GCS, Azure Blob) où les écritures sont immuables et la compaction se fait différemment. RocksDB est embedded dans des dizaines de systèmes (MySQL, TiKV, Kafka interne).

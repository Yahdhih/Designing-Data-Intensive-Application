| Système | Niveau par défaut | Serializable disponible ? | Notes |

### Partie 3 — Arbre de décision « quel niveau d'isolation ? » (15 min)

Dans `reference/notes/jour-061-decision-tree.md` :

```
1. Ai-je des transactions multi-lignes qui doivent être atomiques ?
   Non → utiliser des opérations atomiques (UPDATE x = x + 1) + Read Committed
   Oui → continuer...

2. Puis-je tolérer des lectures non-répétables (lire 2× la même ligne, voir 2 valeurs) ?
   Oui → Read Committed (défaut Postgres)
   Non → Snapshot Isolation (REPEATABLE READ dans Postgres) ou mieux

3. Ai-je des transactions qui lisent un ensemble → décident → écrivent selon la décision ?
   Exemple : médecins de garde, double-booking, username unique
   Non → Snapshot Isolation suffit
   Oui → 2 sous-options :
     a. Sais-tu quelles lignes lire avant d'écrire ?
        Oui → SELECT FOR UPDATE (pessimiste, simple)
        Non → SERIALIZABLE SSI (optimiste, retry nécessaire)

4. Ma charge est-elle très élevée (contention ++) ?
   Oui → éviter SERIALIZABLE 2PL (deadlocks) → préférer SSI ou FOR UPDATE ciblé
   Non → SERIALIZABLE SSI est acceptable

5. Ai-je des phantom reads (requêtes de plage) ?
   Oui → SERIALIZABLE obligatoire (predicate lock)
   Non → Snapshot Isolation + FOR UPDATE peut suffire
```

### Connexion Ch.7 → Ch.8 (10 min)

Ch.7 couvre les transactions sur 1 nœud. Ch.8 introduit les **problèmes des systèmes distribués** :

- Réseau non fiable (messages perdus, retardés, réordonnés)
- Horloges non synchronisées (NTP drift, problème de l'ordre des événements)
- Processus qui peuvent tomber à n'importe quel moment

**Impact sur les transactions :**
- Une transaction ACID sur 1 nœud est relativement simple (WAL + 2PL ou SSI)
- Une transaction ACID sur N nœuds nécessite un **protocole de coordination** (2PC) + **consensus** (Ch.9)
- C'est pourquoi les systèmes NoSQL sacrifient souvent les transactions pour gagner en scalabilité

Note dans `reference/notes/jour-061.md` :
- Pourquoi un `BEGIN ISOLATION LEVEL SERIALIZABLE` sur Postgres n'aide-t-il pas si les données sont sur 2 shards différents ?
- Comment Kafka exactly-once sémantique résout-il un sous-ensemble du problème transactionnel distribué ?

### Cartes de synthèse (5 min)

```tsv
Q: Arbre de décision : quel niveau d'isolation pour éviter le write skew sans using FOR UPDATE ?
R: Utiliser BEGIN ISOLATION LEVEL SERIALIZABLE (SSI dans Postgres ≥ 9.1).
   SSI détecte automatiquement les cycles de dépendances read-write et abort une transaction.
   L'application doit implémenter une logique de retry en cas d'abort SSI.
   Alternative : SELECT FOR UPDATE sur toutes les lignes lues (pessimiste, simple, plus prévisible).
Tags: ch07::transactions::decision

Q: Pourquoi SQLite est-il sérialisable par défaut alors que Postgres utilise Read Committed ?
R: SQLite n'autorise qu'un seul writer à la fois (WAL mode permet 1 writer + N readers).
   → Sérialisation naturelle sans coût additionnel.
   Postgres gère de nombreux clients concurrents → Read Committed par défaut est un compromis
   entre sécurité et débit. Serializable SSI disponible mais plus coûteux.
Tags: ch07::transactions::decision
```

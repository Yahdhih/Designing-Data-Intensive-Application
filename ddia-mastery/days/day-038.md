### Étape 1 — Initialiser le keyspace (5 min)

```bash
docker exec -it ddia-cassandra cqlsh
```

```sql
-- Keyspace avec RF=3 (même si on a 1 nœud — RF max = nœuds disponibles)
-- Pour ce labo avec 1 nœud, utilise RF=1 (toutes les CL deviennent équivalentes)
-- Note : pour observer vraiment QUORUM vs ONE, il faut 3 nœuds (hors-scope ici)
-- → On va simuler les comportements avec des exemples et mesures de latence

CREATE KEYSPACE IF NOT EXISTS ddia
    WITH replication = {'class': 'SimpleStrategy', 'replication_factor': 1};

USE ddia;

CREATE TABLE IF NOT EXISTS users (
    user_id  UUID PRIMARY KEY,
    name     TEXT,
    email    TEXT,
    score    INT
);

-- Insérer des données
INSERT INTO users (user_id, name, email, score)
VALUES (uuid(), 'Alice', 'alice@example.com', 100);
INSERT INTO users (user_id, name, email, score)
VALUES (uuid(), 'Bob', 'bob@example.com', 200);
INSERT INTO users (user_id, name, email, score)
VALUES (uuid(), 'Charlie', 'charlie@example.com', 150);

SELECT * FROM users;
```

### Étape 2 — Observer les consistency levels (15 min)

```sql
-- Consistency ONE : lecture depuis 1 nœud (le plus proche)
CONSISTENCY ONE;
SELECT * FROM users WHERE user_id = ? ;  -- remplace par un UUID observé ci-dessus

-- Consistency QUORUM : lecture depuis ⌈RF/2⌉+1 nœuds
CONSISTENCY QUORUM;
SELECT * FROM users;

-- Consistency ALL : lecture depuis TOUS les nœuds
CONSISTENCY ALL;
SELECT * FROM users;
```

**Observer la différence de comportement (pas de latence ici avec RF=1) :**

```sql
-- Mesure de latence avec TRACING
TRACING ON;

CONSISTENCY ONE;
SELECT * FROM users LIMIT 1;

CONSISTENCY ALL;
SELECT * FROM users LIMIT 1;

TRACING OFF;
```

### Étape 3 — TTL et timestamps (10 min)

```sql
-- Cassandra utilise les timestamps pour LWW (last-write-wins)
-- Insérer avec un timestamp explicite (en µs depuis epoch)

VALUES (uuid(), 'Dave', 'dave@example.com', 50)
USING TTL 60;  -- expire après 60 secondes

-- Voir les timestamps Cassandra (write time)
SELECT user_id, name, WRITETIME(name) FROM users;
```

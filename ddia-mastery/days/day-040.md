### Partie 1 — Matrice de décision (remplir de mémoire, puis vérifier) (15 min)

Dans `reference/tradeoffs.md` — section Ch.5 finale :

**Tableau principal (remplis d'abord sans aide) :**

| Dimension | Mono-leader | Multi-leader | Leaderless |
|-----------|-------------|--------------|------------|
| **Écriture vers** | Leader uniquement | N'importe quel leader | W nœuds quelconques |
| **Cohérence lecture** | Forte (leader) / Éventuelle (follower) | Éventuelle | Éventuelle (W+R>N = cohérence) |
| **Conflits possibles** | Non | Oui | Oui (écritures concurrentes) |
| **Failover** | Nécessaire (+ risqué) | Automatique (autre leader) | Non requis |
| **Disponibilité écriture en panne** | ✗ leader mort → indispo | ✓ autres leaders | ✓ si W nœuds UP |
| **Scalabilité lecture** | ✓ (ajouter followers) | ✓ | ✓ |
| **Scalabilité écriture** | ✗ (bottleneck leader) | ✓ (par datacenter) | ✓ (distribué) |
| **Anomalies typiques** | Lag, RYW, monotonic | Conflits, ordre causal | Lectures périmées, conflits |
| **Complexité opéra.** | Faible | Haute | Moyenne |
| **Exemples** | Postgres, MySQL, Kafka | CockroachDB, Bigtable | Cassandra, Dynamo, Riak |

### Connecter Ch.5 → Ch.6 (10 min)

Dans `reference/notes/jour-040.md` :

La réplication résout : *comment avoir N copies d'un nœud*.
Le partitionnement résout (Ch.6) : *comment diviser les données sur N nœuds*.

En général, les 2 sont utilisés ensemble :
```
Données totales = 10 TB
→ 10 partitions de 1 TB (Ch.6)
→ Chaque partition répliquée 3× (Ch.5)
→ Total = 10 × 3 = 30 nœuds de données
```

**Questions de transition vers Ch.6 :**
1. Si tu partitionnes par `hash(user_id) % N`, que se passe-t-il si N change (tu ajoutes un nœud) ?

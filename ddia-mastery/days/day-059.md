### Étape 1 — FOR UPDATE vs FOR SHARE (10 min)

```sql
-- Setup
CREATE TABLE IF NOT EXISTS inventory (
    product_id INT PRIMARY KEY,
    name       TEXT,
    quantity   INT NOT NULL CHECK (quantity >= 0)
);
INSERT INTO inventory VALUES (1, 'Widget', 100), (2, 'Gadget', 50)
ON CONFLICT DO NOTHING;

-- FOR SHARE : verrou partagé (lecture uniquement)
-- → Plusieurs transactions peuvent avoir FOR SHARE simultanément
-- → Bloque seulement les FOR UPDATE et les UPDATE/DELETE

-- FOR UPDATE : verrou exclusif
-- → Bloque tous les autres FOR UPDATE, UPDATE, DELETE
-- → Autorise les SELECT simples (sans FOR UPDATE) en Read Committed

-- Test simultané (2 sessions) :
-- S1 : BEGIN; SELECT * FROM inventory WHERE product_id = 1 FOR SHARE;
-- S2 : BEGIN; SELECT * FROM inventory WHERE product_id = 1 FOR SHARE;
-- → S2 n'est PAS bloqué (2 FOR SHARE compatibles)

-- S1 : SELECT * FROM inventory WHERE product_id = 1 FOR UPDATE;
-- S2 : SELECT * FROM inventory WHERE product_id = 1 FOR UPDATE;
-- → S2 EST bloqué (FOR UPDATE vs FOR UPDATE incompatibles)
```

**Test à réaliser dans 2 terminaux :**
1. S1 : `BEGIN; SELECT * FROM inventory WHERE product_id = 1 FOR SHARE;`
2. S2 : `BEGIN; SELECT * FROM inventory WHERE product_id = 1 FOR SHARE;` → bloqué ou pas ?
3. S2 : `ROLLBACK;` puis : `BEGIN; SELECT * FROM inventory WHERE product_id = 1 FOR UPDATE;`
4. S2 est bloqué en attendant S1 — `SELECT pid, wait_event_type, query FROM pg_stat_activity WHERE wait_event_type = 'Lock';`
5. S1 : `COMMIT;` → S2 se débloque

### Étape 2 — Provoquer un deadlock (10 min)

```
Session 1                              Session 2
─────────────────────────────────────  ─────────────────────────────────────
BEGIN;
SELECT * FROM inventory
WHERE product_id = 1 FOR UPDATE;
                                       BEGIN;
                                       SELECT * FROM inventory
                                       WHERE product_id = 2 FOR UPDATE;
SELECT * FROM inventory
WHERE product_id = 2 FOR UPDATE;
-- ↑ Bloqué (S2 a le verrou sur product_id=2)
                                       SELECT * FROM inventory
                                       WHERE product_id = 1 FOR UPDATE;
                                       -- ↑ Bloqué (S1 a le verrou sur product_id=1)
                                       -- → DEADLOCK détecté par Postgres !
                                       -- → Une des 2 sessions reçoit :
                                       -- ERROR: deadlock detected
                                       -- DETAIL: Process X waits for ShareLock on
                                       -- transaction Y; blocked by process Z.
```

**Observer dans les logs :**
```bash
docker logs ddia-postgres --tail 20 | grep -i deadlock
```

**Note :** Postgres détecte les deadlocks via un timeout (par défaut 1 seconde avec `deadlock_timeout`). La transaction qui attend le plus longtemps est abortée.

### Étape 3 — NOWAIT et SKIP LOCKED (10 min)

```sql
-- NOWAIT : échoue immédiatement si le verrou n'est pas disponible (pas d'attente)
BEGIN;
SELECT * FROM inventory WHERE product_id = 1 FOR UPDATE NOWAIT;
-- Si verrou occupé → ERROR immédiatement (vs attendre indéfiniment)
-- Utile pour : queue workers (ne pas attendre une tâche déjà prise)
COMMIT;

-- SKIP LOCKED : ignore les lignes verrouillées (les saute)
-- Utile pour : file de travail (task queue)
CREATE TABLE IF NOT EXISTS job_queue (
    job_id     SERIAL PRIMARY KEY,
    status     TEXT DEFAULT 'pending',
    payload    TEXT
);
INSERT INTO job_queue (payload) SELECT 'job-' || i FROM generate_series(1, 10) i;

-- Worker pattern : prendre le prochain job disponible sans attendre
BEGIN;
SELECT * FROM job_queue
WHERE status = 'pending'
ORDER BY job_id
LIMIT 1
FOR UPDATE SKIP LOCKED;
-- → Renvoie 1 job disponible (les jobs déjà pris par d'autres workers sont ignorés)
UPDATE job_queue SET status = 'processing' WHERE job_id = <id_obtenu>;
COMMIT;
```

**Note :** `FOR UPDATE SKIP LOCKED` est le pattern standard pour les file de travail dans Postgres. C'est plus fiable que Redis RPOPLPUSH dans certains cas d'usage.

### Étape 4 — Mesurer l'impact du verrouillage (5 min)

```sql
-- Créer 10k lignes dans inventory pour le benchmark
INSERT INTO inventory (product_id, name, quantity)
SELECT i, 'item-' || i, 100 FROM generate_series(100, 10100) i
ON CONFLICT DO NOTHING;

-- Comparer : SELECT simple vs SELECT FOR UPDATE
EXPLAIN ANALYZE SELECT * FROM inventory WHERE quantity > 50 LIMIT 100;
EXPLAIN ANALYZE SELECT * FROM inventory WHERE quantity > 50 LIMIT 100 FOR UPDATE;
-- → FOR UPDATE nécessite l'acquisition de verrous → légèrement plus lent
-- → Observer le temps d'exécution dans "Planning time" + "Execution time"
```

### Passe 3 — Corps sélectif (22 min)

**Priorité 1 — Section 3 : Critique des phénomènes ANSI (7 min)**

Les définitions ANSI sont basées sur des « phénomènes » (anomalies) interdits à chaque niveau. Mais les auteurs montrent que ces définitions sont **ambiguës** :
- Les phénomènes sont définis en termes d'historiques spécifiques, pas d'historiques généraux
- Cela laisse des comportements indésirables qui ne correspondent à *aucun* phénomène ANSI

Comprendre : qu'est-ce qui manque dans P1/P2/P3 pour couvrir les lost updates et write skews ?

**Priorité 2 — Section 4 : Nouveaux phénomènes (10 min)**

Les auteurs ajoutent :
- **P0 (Dirty Write)** : 2 txns écrivent sur la même donnée avant de commiter — manquait dans ANSI
- **P4 (Lost Update)** : clairement défini formellement (n'était qu'implicite dans ANSI)
- **A5A (Read Skew)** : T1 lit A, T2 modifie A et B et commite, T1 lit B → incohérence (différent de write skew !)
- **A5B (Write Skew)** : le problème des médecins — explicitement formalisé ici pour la première fois

**Note historique :** La formalisation du write skew dans ce papier est la raison pour laquelle DDIA peut le décrire précisément. Ce papier a introduit le terme « write skew ».

**Priorité 3 — Section 5 : Isolation levels revisited (5 min)**

Le papier propose un nouveau niveau : **Cursor Stability** (entre Read Committed et Repeatable Read).  
Il propose aussi de définir les niveaux par rapport aux *historiques admis*, pas aux *phénomènes interdits*.

Note la connexion avec Snapshot Isolation : elle n'est pas dans ANSI SQL mais le papier la mentionne.

### 2 cartes Anki (5 min)

```tsv
Q: Pourquoi les définitions ANSI SQL des niveaux d'isolation sont-elles défectueuses selon Berenson et al. (1995) ?
R: Les phénomènes ANSI (P1=dirty read, P2=non-repeatable read, P3=phantom) ne couvrent pas
   toutes les anomalies possibles.
   Notamment : P0 (dirty write), P4 (lost update), et A5B (write skew) sont absents.
   Les définitions sont aussi ambiguës car basées sur des historiques spécifiques.
Tags: ch07::transactions::articles::critique-ansi

Q: Quel niveau d'isolation garantit à la fois l'absence de write skew ET de phantom reads ?
R: Serializable (le seul niveau qui interdit TOUTES les anomalies).
   Snapshot Isolation (Repeatable Read) ne suffit pas pour le write skew (A5B dans Berenson).
   → Utiliser BEGIN ISOLATION LEVEL SERIALIZABLE dans Postgres (SSI depuis Postgres 9.1).
Tags: ch07::transactions::articles::critique-ansi
```

### Points clés à retenir

1. **B-tree :**
   - Pages de taille fixe (4KB typiquement) → stockage sur disque naturel
   - Mise à jour **en place** → pas besoin de compaction, mais…
   - WAL obligatoire : chaque écriture → d'abord dans le WAL → puis dans l'arbre
   - Crash pendant une mise à jour → reconstruire depuis le WAL
   - Optimisé pour les **lectures** (une seule descente d'arbre O(log n))

2. **LSM-tree vs B-tree — le résumé :**
   - LSM → écritures séquentielles rapides · lectures potentiellement lentes
   - B-tree → lectures rapides · écritures avec amplification (WAL + page rewrite)
   - Règle pratique : write-heavy → LSM · read-heavy / range-scans → B-tree

3. **OLTP vs OLAP :**
   - **OLTP** (*Online Transaction Processing*) : requêtes courtes, transactions, accès par clé (Postgres, MySQL) → optimisé pour la latence
   - **OLAP** (*Online Analytical Processing*) : agrégations sur des milliards de lignes, quelques colonnes, batch → optimisé pour le throughput

4. **Stockage en colonnes :**
   - Stocker toutes les valeurs d'une colonne ensemble (vs une ligne entière ensemble)
   - `SELECT AVG(prix) FROM ventes` → ne lit que la colonne `prix`, pas `client_id`, `adresse`…
   - Compression plus efficace : même type → même codec · compression par dictionnaire
   - Exécution vectorisée : opérations SIMD sur des tableaux contigus
   - **Parquet, ORC** — c'est ce que Spark lit → lien avec ton expérience !

### 📌 Contexte 2026 (si 1ʳᵉ éd.)
La 2ᵉ éd. souligne que le stockage objet (S3) est devenu la couche de stockage dominante pour l'OLAP. Les formats Parquet/ORC sur S3 + des moteurs comme DuckDB, BigQuery, ou Spark sont la norme. La distinction OLTP/OLAP tient mais les architectures cloud-native la réalisent différemment (« lakehouse »).

## 📖 Lecture — 35 min

📌 **Localise la section :** `reference/toc.md` → Ch.5 "Réplication sans leader" / "Leaderless Replication"

**1. Architecture sans leader (5 min)**

- Pas de failover de leader : n'importe quel nœud accepte les écritures
- Le client (ou un coordinateur) envoie les écritures à W nœuds en parallèle
- Le client lit depuis R nœuds en parallèle et prend la valeur avec la version la plus haute
- Exemple de systèmes : Amazon Dynamo · Cassandra · Riak · Voldemort

**2. Quorums W + R > N (10 min)**

La garantie fondamentale :
```
N = 3, W = 2, R = 2 → W + R = 4 > 3 ✓
  Nœuds écrits : {n1, n2}  ou  {n1, n3}  ou  {n2, n3}
  Nœuds lus  : {n1, n2}  ou  {n1, n3}  ou  {n2, n3}
  → Intersection garantie ≥ 1 nœud → au moins 1 nœud a la valeur récente
```

Configurations courantes :
- `N=3, W=2, R=2` : tolérance à 1 panne (Cassandra QUORUM)
- `N=5, W=3, R=3` : tolérance à 2 pannes
- `N=3, W=3, R=1` : écriture forte, lecture légère
- `N=3, W=1, R=3` : écriture légère, lecture forte (Cassandra ONE write + ALL read)

Limites des quorums :
- LWW peut supprimer des écritures concurrentes (même problème que multi-leader)
- « Sloppy quorum » peut temporairement violer W+R>N

**3. Sloppy quorum + hinted handoff (10 min)**

**Sloppy quorum :** Si les W nœuds habituels sont indisponibles, le client écrit sur W nœuds *alternatifs* (hors de la partition normale). Avantage : disponibilité en écriture. Inconvénient : W+R>N n'est plus garanti pendant la partition réseau.

**Hinted handoff :** Les nœuds alternatifs stockent les écritures avec une *hint* (note) indiquant leur destination réelle. Quand le nœud habituel redevient disponible, les données lui sont transférées.

*Exemple :* Cassandra avec `consistency_level=ANY` utilise le sloppy quorum → haute disponibilité en écriture, mais lecture peut être périmée.

**4. Détecter les écritures concurrentes + versions (10 min)**

Deux clients écrivent la même clé simultanément → conflit. Comment détecter qui a écrit en premier si les horloges ne sont pas fiables ?

Solutions :
- **Version vectors** (vector clocks Dynamo) : chaque nœud maintient un compteur `[n1:3, n2:1]` → on peut détecter l'ordre causal
- **Read repair** : lors d'une lecture qui détecte des valeurs périmées sur certains nœuds → le client répare ces nœuds
- **Anti-entropy background process** : processus de fond qui compare et synchronise les nœuds

### Connexion avec Cassandra (10 min)

Dans Cassandra, les « consistency levels » correspondent directement à W et R :

| Cassandra CL | Correspondance quorum | Quand l'utiliser |
|---|---|---|
| `ONE` | W=1 ou R=1 | Haute disponibilité, cohérence éventuelle |
| `QUORUM` | W=⌈N/2⌉+1 ou R=⌈N/2⌉+1 | Compromis disponibilité/cohérence |
| `ALL` | W=N ou R=N | Cohérence forte, aucune tolérance panne |
| `LOCAL_QUORUM` | QUORUM dans le datacenter local | Multi-datacenter, écriture locale |

**Question :** Pourquoi `QUORUM` write + `QUORUM` read garantit-il W+R>N avec un RF=3 (replication factor 3) ?
